# Strata Ingest Activity Log Plan

Status: browser/read-only slice present with a SQLite run-list projection.

This plan covers a trace-backed activity log for source ingestion and raw-to-wiki organization. It is subordinate to [roadmap.md](./roadmap.md), [wiki-plan.md](./wiki-plan.md), and [web-control-plane-plan.md](./web-control-plane-plan.md).

## Objective

Expose a local, inspectable activity log that answers:

- What source content did Strata ingest?
- Was it written, skipped, failed, or only previewed in a dry-run?
- Which raw snapshot was created or read?
- Which curated wiki pages were created or updated?
- Which people, projects, decisions, threads, and action lists did the item affect?
- Which session trace explains the operation?

The feature should make ingestion auditable without turning `wiki/log.md` into a machine database. `wiki/log.md` remains the compact human chronology; structured activity comes from durable session events and traces.

## Product Shape

The first useful surface should be an `Activity` or `Ingest` view in the local web app. It should show recent ingest runs grouped by session, with expandable per-source items.

A run summary should include:

- Session id, title, status, started/ended timestamps, source connector, operation, and dry-run/apply mode.
- Counts for raw snapshots written, raw sources scanned, sources indexed, sources skipped, failures, and wiki pages touched.
- Links to the associated session trace and relevant connector/raw-to-wiki command context.

A per-source item should include:

- Source type: Granola, Notion, Slack, or future connector.
- Stable source id and title when available.
- Raw path, if a raw snapshot exists.
- Organization result: primary wiki page plus touched people, projects, decisions, threads, and action lists.
- Written paths and skipped/failed reason.
- Whether the item was generated by connector pull, raw-to-wiki indexing, or both.

The browser should link activity items to the existing wiki browser for curated pages and to raw/source paths when those are intentionally visible.

## Architecture

The source of truth is the existing local session system:

- `SessionStore` persists sessions and events in `.strata/state.sqlite` and JSONL traces under `.strata/traces/`.
- Connector operations already run as `kind: "ingest"` sessions through the shared connector runner.
- Raw-to-wiki indexing already records `raw_to_wiki.index.*` lifecycle events.

Do not parse `wiki/log.md` to build the activity feed. Do not put raw-to-wiki classification or retrieval-ranking logic in the browser.

The backend activity service lives in `@strata/ingest/activity` so CLI and web callers can share one contract. `packages/web-api/src/activityServices.ts` is a thin adapter over that service. The service keeps `ingest_activity_runs` as a local SQLite run-list projection populated from the same append-only session events. List filters use projection columns such as source, start time, raw writes, wiki indexes, search indexes, skips/previews, and failures; detail views still read the canonical event timeline.

Expose the service through tRPC, for example:

```ts
activity: t.router({
  list: t.procedure.input(activityListInput).query(...),
  get: t.procedure.input(activitySessionInput).query(...),
})
```

Optional later CLI support can reuse the same service shape:

```bash
bun run strata ingest activity --limit 50
bun run strata ingest activity --source slack --json
```

## Implementation Status

The first read-only milestone is implemented:

- `@strata/ingest/activity` normalizes recent job, connector, and raw-to-wiki sessions into run summaries plus item timelines.
- Connector runner traces now include `connector.<source>.<operation>.item` and `.failure` events in addition to aggregate completion events.
- Raw-to-wiki traces now include complete organization metadata on indexed items and explicit skipped-item events.
- Scheduled job traces now include schedule id/name metadata and redacted job input for future debugging.
- `ingest_activity_runs` materializes recent run summaries from session events so `/activity` can filter by source and selected result categories without scanning a limited session window.
- `activity.list` and `activity.get` expose the normalized DTOs through tRPC.
- `/activity` shows recent runs with source filters, status/counts, trace paths, schedule/parent-job correlation, and expandable item rows.

Remaining planned work: add a CLI activity view if useful, add richer filters/search/proposal links, link activity rows directly into wiki pages once the wiki browser supports deeplinks, and consider item-level/search projections only if detail timelines become too expensive.

## Event Contract

Prefer small append-only event payloads over reconstructing behavior from written Markdown.

Connector sessions should provide enough events to answer:

- operation started/completed/failed;
- connector name and operation;
- redacted config summary;
- raw snapshot items written/skipped/failed;
- stable source ids, titles, raw paths, and statuses.

Raw-to-wiki sessions should provide enough events to answer:

- scan started/completed;
- each indexed item;
- each skipped item with reason;
- primary curated page path;
- touched people/project/decision/thread/action paths;
- written paths;
- dry-run/apply mode.

Current `raw_to_wiki.index.item` events already contain part of this shape. The implementation should expand those payloads to include the same organization metadata returned by raw-to-wiki result items, and add explicit skipped-item events for useful diagnostics.

All payloads must avoid secrets. Connector configs and errors must continue using existing redaction helpers before events are persisted or returned to the browser.

## Sequencing

1. Inventory current ingest event payloads and add tests that lock the existing `connector.*` and `raw_to_wiki.index.*` event shapes. Status: complete for representative connector/raw-to-wiki events.
2. Add an activity normalization service over `SessionStore` sessions/events with DTO tests using fixture event streams. Status: complete in `@strata/ingest/activity`.
3. Expand raw-to-wiki instrumentation so indexed and skipped items carry complete organization metadata. Status: complete.
4. Add connector item-level events where pull results currently only expose aggregate completion payloads. Status: complete.
5. Add tRPC procedures for listing recent activity runs and fetching one run's detailed item timeline. Status: complete.
6. Add the web `Activity`/`Ingest` route with run summaries, filters, expandable items, and links into `/wiki` and session traces. Status: route present with trace paths; direct wiki deeplinks remain future work.
7. Materialize run-list activity rows in SQLite for faster filtering over large backfills while keeping append-only session events as the canonical record. Status: complete for run summaries.
8. Optionally add `strata ingest activity` as a CLI view over the same normalized service.

## Acceptance Criteria

The first milestone is complete when:

- The web app can show recent ingest sessions without reading `wiki/log.md`.
- Each run shows status, source, operation, dry-run/apply mode, counts, and trace/session link.
- Raw-to-wiki runs show each indexed source and where it was organized in the wiki.
- Skipped sources show clear reasons, especially low-signal Slack skips and already-indexed raw files.
- Connector pull runs show raw snapshot paths and item statuses when available.
- Browser responses and persisted events do not expose connector secrets.
- Tests cover activity normalization and representative connector/raw-to-wiki event payloads.

A later milestone can add richer filters, full-text search over activity items, proposal links, schedule-run correlation, and materialized indexing for large histories.
